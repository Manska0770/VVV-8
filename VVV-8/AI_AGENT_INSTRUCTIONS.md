# AI Agent Instructions for VVV-8

## Цель
Поддерживать и развивать клиентский статический сайт знакомств в каталоге `VVV-8`.

## Текущая ситуация
- Развернутая версия сайта на `https://vvv-8.onrender.com` сейчас отдаёт корневой `index.html`, а не содержимое из папки `VVV-8`.
- Файл `vvv-config.js` на развернутом хосте возвращает 404, поэтому runtime-конфигурация Supabase не загружается.
- В результате профиль не сохраняется.
- Нужно не только исправить логику внутри `VVV-8`, но и убедиться, что деплой действительно публикует `VVV-8` и что `vvv-config.js` доступен.

## Основная задача
- Хранение данных должно использовать единый объект `vvv_database` в `localStorage`.
- Реальные анкеты пользователей должны сохраняться в `vvv_database.profiles` и участвовать в поисковой выдаче.
- Старые ключи (`vvv_user`, `profileName`, `profileAge`, `profileCity`, `profileBio`, `profileAvatar`, `profileGender`, `profilePhoto`, `profileGoal`, `vvv_search_filters`, `vvv_likes`, `vvv_matches`, `vvv_chat_*`, `vvv_chat_mutual_*`) поддерживаются только для миграции.
- Новые данные должны сохраняться и читаться из `vvv_database`.

## Структура `vvv_database`
- `user` — объект профиля текущего пользователя
- `profiles` — массив кандидатов (текущие тестовые профили)
- `likes` — массив ID профилей, которые пользователь лайкнул
- `matches` — массив ID профилей, с которыми есть взаимный лайк
- `chats` — объект вида `{ [id]: [messages] }`
- `mutualChats` — объект вида `{ [id]: true }`
- `notifications` — массив уведомлений
- `filters` — объект текущих фильтров поиска

## Что исправлять
- Удалять дублирование legacy-ключей в новых функциях.
- Сохранять пользовательский профиль и фильтры только через `vvv_database`.
- Поддерживать fallback на legacy-ключи только при первой миграции.
- Не оставлять прямых `localStorage.getItem('vvv_*')` и `localStorage.setItem('vvv_*')` вызовов в новых рабочий коде, за исключением миграции.
- Добавлять новые helper-функции в `VVV-8/app.js`, `VVV-8/chat.js` и `VVV-8/notifications.js` для работы с базой.

## Какие страницы редактировать
- `VVV-8/app.js` — основное приложение, фильтры, профиль, сохранение
- `VVV-8/chat.js` — история чатов и mutual-статус
- `VVV-8/notifications.js` — уведомления и реальные матчи
- `VVV-8/profile.html` — отображение профиля пользователя

## Тесты и валидация
- Проверять синтаксис файлов командой `node --check VVV-8/app.js VVV-8/chat.js VVV-8/notifications.js`
- Проверять, что `vvv_database` создаётся и читается без ошибок
- Проверять, что старые ключи используются только в миграционной функции
- Проверять, что Supabase-поля `SUPABASE_URL` и `SUPABASE_ANON_KEY` остаются пустыми в репозитории, если не настроены через внешний конфиг

## Supabase и удалённое хранилище
- `VVV-8/app.js` должен поддерживать динамическую загрузку `@supabase/supabase-js` из CDN
- При старте приложения `init()` запускает `initSupabase()` и `syncRemoteProfilesToLocal()` при наличии настроенных ключей
- При сохранении профиля `addOrUpdateDatabaseProfile()` должен обновлять `vvv_database.profiles` и отправлять профиль в Supabase, если клиент доступен
- Если Supabase недоступна или ключи не заданы, приложение должно работать локально без ошибок

## Дополнительное правило
Если добавляешь новый persistent state, дай ему явное имя в `vvv_database`, чтобы данные было легко мигрировать и переносить в будущем.

## Render и деплой
- Render разворачивает статический сайт из папки `VVV-8` при ветке `main`.
- Основной Render конфиг находится в `render.yaml` в корне репозитория.
- GitHub workflow для запуска деплоя — `.github/workflows/deploy-to-render.yml`.
- Если меняешь структуру файлов или путь публикации, обнови `publishPath` в `render.yaml` и убедись, что `branch: main` остаётся актуальной.
- Все изменения пользовательского приложения должны вноситься в папке `VVV-8`:
  - `VVV-8/app.js`
  - `VVV-8/chat.js`
  - `VVV-8/notifications.js`
  - `VVV-8/profile.html`
- Не изменяй рабочие файлы вне `VVV-8`, если только не требуется обновить Render-конфигурацию или workflow.
- После правок проверяй, что сборка/синтаксис проходят, и что `render.yaml` по-прежнему указывает на `VVV-8`.

## Пример рабочего процесса
1. Вноси изменения в файлы внутри `VVV-8`:
   - `VVV-8/app.js`
   - `VVV-8/chat.js`
   - `VVV-8/notifications.js`
   - `VVV-8/profile.html`
2. Проверяй синтаксис:
   - `cd /workspaces/VVV-8/VVV-8 && node --check app.js chat.js notifications.js`
3. Фиксируй изменения в git и пушь в ветку `main`.
4. Render автоматически задеплоит новую версию из `main` на основании `render.yaml`.
5. Если необходимо, вручную запусти workflow `Deploy to Render` или проверяй, что `push` на `main` сработал.

## Что не менять без необходимости
- Не изменяй файлы вне `VVV-8`, если задача не связана с настройкой Render.
- Не меняй `publishPath` и `branch` в `render.yaml` без ясной причины.
